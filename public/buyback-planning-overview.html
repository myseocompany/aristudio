AriStudio • Planning • BuybackSystem

Así va a funcionar el nuevo subsistema de planeación.

El sistema no nace como agenda automática. Primero ordena datos, clasifica trabajo y deja evidencia clara sobre qué debe seguir haciendo Nicolas, qué debe documentarse, qué debe delegarse y qué soporte externo conviene resolver primero.

Fase 1

Datos, servicios, seeds y reglas DRIP. Sin API pública, sin MCP nuevo y sin protección real de calendario.

4 loops

Audit, Transfer, Fill y Review. El sistema no termina al clasificar: obliga a revisar outcomes.

Idea central

1. Decide con datos

Reutiliza tareas, usuarios, proyectos, estados, puntos del timer y relaciones reales del repo antes de inventar campos duplicados.

2. Separa intención y ejecución

En Fase 1 puede haber preferencias y sugerencias. La protección efectiva del calendario llega después, cuando exista modelo verificable.

3. Fuerza salida operativa

Si algo drena o depende demasiado de Nicolas o de Luisa, el sistema debe dejar una acción visible de salida, no solo una etiqueta.

Cómo fluye una tarea

Paso 1

Se detecta

La tarea entra si es planificable según su estado real, responsable, puntos, proyecto y contexto operativo.

Paso 2

Se clasifica

El servicio Buyback calcula DRIP, acción recomendada, riesgo de continuidad y prioridad contextual.

Paso 3

Se decide salida

Si la tarea drena o se repite, genera sugerencias persistidas: playbook, delegación, soporte o revisión.

Paso 4

Se revisa

Después de delegar o documentar, el sistema revisa outcome, actualiza playbooks y recalibra prioridades.

Matriz DRIP

La pregunta no es solo si la tarea es urgente. La pregunta es si genera valor y si energiza o drena.

Bajo valor + drena Delegation

No debería entrar al día de Nicolas salvo emergencia. El sistema debe empujarla fuera del operador principal.

Alto valor + drena Replacement

Puede seguir entrando temporalmente, pero obliga a crear salida: SOP, soporte, automatización o rediseño.

Bajo valor + energiza Investment

Puede quedarse si no desplaza trabajo de más valor y si ayuda a construir capacidades futuras.

Alto valor + energiza Production

Es donde Nicolas debería pasar la mayor parte del tiempo. Aquí se protege el foco estratégico.

Entidades principales

Auditoría

BuybackAudit

Guarda la clasificación DRIP, scores, acción sugerida y huella de idempotencia para no duplicar auditorías iguales.

Delegación

DelegationCandidate

Representa trabajo que debería salir del operador principal, con urgencia, documentación y tipo de dueño recomendado.

Apoyo

SupportNeed

Describe la necesidad real de soporte por área funcional. No reutiliza la tabla de roles RBAC.

Sistema

Playbook

Convierte trabajo repetido en checklist, SOP, assets y documentación delegable conectada a un área de soporte.

Visibilidad

PlanningSuggestion

Evita sugerencias invisibles. Si el sistema recomienda algo, queda persistido y consultable.

Continuidad

DelegationOutcome

Registra si una delegación funcionó, falló, volvió al operador o exige actualizar el playbook.

Reglas de contexto que el sistema debe respetar

Laura

  • No aparece como opción asignable dentro del subsistema.
  • Se conserva histórico si existe, pero nunca se recomienda como responsable futuro.
  • Las tareas abiertas heredadas deben quedar marcadas para reasignación.

Luisa

  • Se trata como salida probable y riesgo de continuidad.
  • No debe quedar a cargo de procesos críticos.
  • Sus dependencias deben activar transición y revisión prioritaria.

COOTILCA

  • Entra como proyecto estratégico.
  • Puede elevar prioridad, pero no debería comerse todo el calendario.
  • Su dependencia directa de Nicolas pesa fuerte en la recomendación de soporte.

Calendario

  • Fase 1 solo maneja preferencias y reglas declarativas.
  • No existe aún protección real de bloques familiares o personales.
  • Eso queda para `protected_blocks` o integración externa en Fase 4.

Qué entrega cada fase

Base
01

Fase 1: datos y dominio

  • Tablas de planning, settings, sugerencias y outcomes.
  • Servicios compartidos para clasificar, auditar y recomendar soporte.
  • Seeds de soporte, COOTILCA y configuración del operador principal.
  • Tests del dominio y reglas de continuidad.
Operación
02

Fase 2: MCP especializado

  • Servidor MCP separado para planning Buyback.
  • Tools para auditar, clasificar, listar candidatos y recomendar primer soporte.
  • Permisos por módulos y salidas estructuradas reutilizando los mismos servicios.
API
03

Fase 3: endpoints

  • API versionada para planning, con Form Requests y Resources.
  • Listados con límites, filtros y contratos consistentes con MCP.
  • Métricas Buyback para consumo interno.
Agenda
04

Fase 4: protección real de calendario

  • Bloques protegidos o integración con calendario externo.
  • Revisión histórica por versión del framework.
  • UI administrativa para pesos, umbrales y ladder de soporte.

Ejemplo práctico

Tarea repetida

“Seguimiento administrativo COOTILCA” aparece tres veces en 30 días. El sistema la detecta como repetida, drena y tiene bajo valor económico.

Decisión

La clasifica como `Delegation`, crea sugerencia persistida, propone `SupportNeed` de project management y deja candidato de delegación.

Salida

Cuando alguien asume esa función, el sistema revisa outcome. Si falla, devuelve la tarea, actualiza el playbook y recalcula la recomendación.

Resumen ejecutivo

AriStudio Planning no es un planner decorativo. Es una capa de decisión que transforma tareas reales del repo en evidencia operativa: qué debe sostener Nicolas, qué debe salir de él, qué soporte hace falta primero y qué procesos necesitan documentación para no depender de una sola persona.
