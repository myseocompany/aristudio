Arquitectura Técnica

Cómo está pensada la propuesta por dentro.

Este diagrama narrado está orientado a un programador. La idea es mostrar capas, tablas, servicios, puntos de entrada y orden de implementación sin quedarse en lenguaje de negocio.

Principio

El dominio vive primero

La lógica no debe nacer en controllers ni en tools MCP. Primero se modelan datos y servicios; después se agregan adaptadores de entrada y salida.

Objetivo técnico

Evitar duplicación

API futura, MCP y tests deben usar exactamente los mismos servicios de dominio para que no existan dos implementaciones del clasificador o del recomendador.

Capas de la arquitectura

Entrada

Interfaces

MCP tools en Fase 2 y API/HTTP en Fase 3. Solo validan input, autorizan y serializan output.

Modelos + tablas

`buyback_audits`, `delegation_candidates`, `support_needs`, `playbooks`, `planning_suggestions`, `delegation_outcomes`.

Regla clave

Los adaptadores no piensan

Si una MCP tool o un controller necesita saber si una tarea es `Delegation` o `Replacement`, no debe reimplementar la regla. Debe llamar al servicio de dominio.

Tablas nuevas y responsabilidad de cada una

Catálogo `support_areas`

Tabla maestra funcional. Evita enums rígidos y sirve como punto de unión para delegación, soporte y playbooks.

Snapshot `buyback_audits`

Persistencia del resultado DRIP. Guarda sujeto, auditor, scores, cuadrante, acción y `audit_fingerprint`.

Backlog `delegation_candidates`

Trabajo que debería salir del operador principal. Todavía no implica contratación o automatización ejecutada.

Need `support_needs`

Describe la necesidad de apoyo real. No toca `roles` de RBAC y puede nacer de varios candidatos.

SOP `playbooks`

Documentación operativa reusable. Normalmente nace desde trabajo repetido o desde `Replacement`.

Visibilidad `planning_suggestions`

Hace visibles las recomendaciones del motor. Si no existe esta capa, el sistema “piensa” pero nadie lo ve.

Review `delegation_outcomes`

Registra qué pasó después de delegar: funcionó, falló, volvió al operador o pide actualizar playbook.

Config `planning_settings`

Resuelve `operator_user_id` y evita hardcodear a Nicolas por nombre, email o id.

Extensión `users`

Se amplía con `exit_status`, `is_assignable`, `continuity_risk`, `needs_transition`, `transition_notes`.

Relaciones conceptuales

support_areas ├─ hasMany support_needs ├─ hasMany delegation_candidates └─ hasMany playbooks users ├─ hasMany buyback_audits as subject_user_id ├─ hasMany buyback_audits as auditor_user_id ├─ hasMany delegation_candidates as current_owner_user_id └─ owns planning records via owner_user_id tasks ├─ hasMany buyback_audits ├─ hasMany delegation_candidates └─ hasOne/Many playbooks (por related_task_id) projects ├─ hasMany buyback_audits ├─ hasMany delegation_candidates └─ hasMany playbooks delegation_candidates ├─ belongsTo support_area ├─ belongsTo support_need (nullable) └─ hasMany delegation_outcomes

Servicios y responsabilidad puntual

Servicio 1

`PlanningContextService`

  • Resolver `operator_user_id`.
  • Excluir a Laura del universo asignable.
  • Calcular dependencia de Luisa.
  • Definir `isTaskPlannable()`.
Servicio 2

`BuybackClassifier`

  • Recibir scores y contexto.
  • Determinar cuadrante DRIP.
  • Emitir `recommended_action`.
  • Generar `classification_hash`.
Servicio 3

`BuybackAuditService`

  • Persistir auditorías idempotentes.
  • Construir `audit_fingerprint`.
  • Versionar cambios de clasificación.
  • Disparar sugerencias de salida.
Servicio 4

`DelegationRecommendationService`

  • Puntuar `support_areas`.
  • Recomendar primer soporte.
  • Explicar factores y pesos usados.
  • Priorizar continuidad y bloqueos estratégicos.

Flujo de ejecución

1
Entrada

Una task entra por seed, proceso batch, MCP o API futura.

2
Contexto

`PlanningContextService` valida si es planificable y qué restricciones contextuales aplican.

3
Clasificación

`BuybackClassifier` devuelve DRIP y acción recomendada.

4
Persistencia

`BuybackAuditService` guarda snapshot, crea sugerencias y evita duplicados.

5
Delegación

Si aplica, se crea o sugiere `delegation_candidate` y/o `support_need`.

6
Playbook

Si el trabajo es repetido o crítico, se propone documentación operativa.

7
Review

Más adelante, `delegation_outcomes` mide si la salida funcionó.

8
Exposición

MCP o API leen el mismo estado persistido y no recalculan lógica por su cuenta.

Puntos de entrada técnicos

Fase 2

MCP server

`BuybackPlannerServer` expone tools como `RunBuybackAudit`, `ClassifyTaskBuyback`, `ListSupportNeeds` y `MarkUserAsPlanningExit`.

Fase 3

API

Endpoints `/api/v1/planning/*` usarían Form Requests, Resources y exactamente los mismos servicios internos.

Request -> Authorization -> Validation -> Service -> Eloquent Models -> Response::structured / API Resource

Orden recomendado de implementación

1. Migraciones + modelos 2. support_areas seeder + planning_settings 3. PlanningContextService 4. BuybackClassifier 5. BuybackAuditService 6. DelegationRecommendationService 7. Tests unitarios de dominio 8. MCP tools 9. API versionada 10. protected_blocks / calendario real

Qué no hace la Fase 1

No

No agenda automáticamente

Puede clasificar y sugerir, pero no proteger bloques reales de calendario todavía.

No

No mezcla dominio con UI

La pantalla puede llegar después. La primera inversión es en reglas y persistencia confiables.

No

No duplica lógica

Ni MCP ni API deben tener forks del clasificador o del recomendador.
